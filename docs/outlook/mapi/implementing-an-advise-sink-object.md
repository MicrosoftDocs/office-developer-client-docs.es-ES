# <a name="implementing-an-advise-sink-object"></a><span data-ttu-id="0d62b-103">Implementar un objeto receptor de sugerencias</span><span class="sxs-lookup"><span data-stu-id="0d62b-103">Implementing an Advise Sink Object</span></span>

  
  
<span data-ttu-id="0d62b-104">**Hace referencia a**: Outlook</span><span class="sxs-lookup"><span data-stu-id="0d62b-104">**Applies to**: Outlook</span></span> 
  
<span data-ttu-id="0d62b-105">Un cliente puede implementar sus propios objetos de receptor advise o utilizar una función de utilidad, [HrAllocAdviseSink](hrallocadvisesink.md).</span><span class="sxs-lookup"><span data-stu-id="0d62b-105">A client can either implement its own advise sink objects or use a utility function, [HrAllocAdviseSink](hrallocadvisesink.md).</span></span> <span data-ttu-id="0d62b-106">**HrAllocAdviseSink** crea un objeto de receptor advise con una implementación de **OnNotify** que invoca una función de devolución de llamada.</span><span class="sxs-lookup"><span data-stu-id="0d62b-106">**HrAllocAdviseSink** creates an advise sink object with an implementation of **OnNotify** that invokes a callback function.</span></span> 
  
<span data-ttu-id="0d62b-107">Existen ventajas y desventajas al uso de **HrAllocAdviseSink**.</span><span class="sxs-lookup"><span data-stu-id="0d62b-107">There are advantages and disadvantages to using **HrAllocAdviseSink**.</span></span> <span data-ttu-id="0d62b-108">Puede guardar el trabajo, pero no proporciona ningún control sobre el objeto de receptor advise que crea de recuento de referencias.</span><span class="sxs-lookup"><span data-stu-id="0d62b-108">It can save work, but it provides no control over reference counting the advise sink object that it creates.</span></span> <span data-ttu-id="0d62b-109">Por lo tanto, deben construir su propia implementación de **IMAPIAdviseSink** y evitar el uso de **los clientes que necesitan controlar cuidadosamente la versión del receptor de sus notificaciones o que tienen interdependencias entre su receptor de notificaciones y otro objeto de cliente HrAllocAdviseSink** por completo.</span><span class="sxs-lookup"><span data-stu-id="0d62b-109">Therefore, clients that need to carefully control their advise sink's release or that have interdependencies between their advise sink and another client object should construct their own **IMAPIAdviseSink** implementation and avoid using **HrAllocAdviseSink** altogether.</span></span> 
  
<span data-ttu-id="0d62b-110">Un cliente implementar su propio receptor de notificaciones hacen que un objeto independiente no relacionados a o dependen de otros objetos con el fin de eliminar posibles complicaciones en versión de objeto y el recuento de referencia.</span><span class="sxs-lookup"><span data-stu-id="0d62b-110">A client implementing its own advise sink should make it an independent object not related to or dependent upon any other objects so as to eliminate potential complications in reference counting and object release.</span></span> <span data-ttu-id="0d62b-111">Sin embargo, si debe implementar su receptor de notificaciones como parte de otro objeto o incluir un puntero a otro objeto como un miembro de datos, se recomienda que se mantienen dos recuentos de referencia independientes: uno para el objeto al que hace referencia el receptor de notificaciones y otro para el receptor de aviso.</span><span class="sxs-lookup"><span data-stu-id="0d62b-111">However, if you must implement your advise sink as part of another object or include a back pointer to another object as a data member, it is recommended that two separate reference counts be maintained: one for the object referenced by the advise sink and one for the advise sink.</span></span> 
  
<span data-ttu-id="0d62b-112">Cuando el recuento de referencia del objeto que se hace referencia llega a cero, todos sus métodos pueden producirse un error y se puede destruir su tabla vtable, pero la memoria para el receptor de notificaciones debe permanecer intacta hasta después de que su recuento de referencia también entra en cero.</span><span class="sxs-lookup"><span data-stu-id="0d62b-112">When the reference count of the referenced object falls to zero, all of its methods can fail and its vtable can be destroyed, but the memory for the advise sink must remain intact until after its reference count also falls to zero.</span></span> <span data-ttu-id="0d62b-113">Esto significa que debe reducir su recuento de referencia y de fin destruir el objeto cuando ese recuento llega a cero (método) **versión** del receptor de notificaciones.</span><span class="sxs-lookup"><span data-stu-id="0d62b-113">This means that the advise sink's **Release** method must decrement its reference count and finish destroying the object when that count reaches zero.</span></span> <span data-ttu-id="0d62b-114">Si no se mantienen dos recuentos de referencia independiente, sería fácil de destruir accidentalmente el receptor de notificaciones como parte del proceso de **versión** del objeto que abarca.</span><span class="sxs-lookup"><span data-stu-id="0d62b-114">If two separate reference counts are not maintained, it would be easy to inadvertently destroy the advise sink as part of the encompassing object's **Release** process.</span></span> 
  
<span data-ttu-id="0d62b-115">Uso de **HrAllocAdviseSink** para implementar un receptor de notificaciones de clientes deben ser igualmente cuidados de no incluir su función de devolución de llamada como un método en otro objeto de receptor advise.</span><span class="sxs-lookup"><span data-stu-id="0d62b-115">Clients using **HrAllocAdviseSink** to implement an advise sink must be equally careful not to include their callback function as a method in another advise sink object.</span></span> <span data-ttu-id="0d62b-116">Para los clientes de C++, es tentador hacerlo y pasar el puntero _this_ como un parámetro.</span><span class="sxs-lookup"><span data-stu-id="0d62b-116">For C++ clients, it is tempting to do this and pass the  _this_ pointer as a parameter.</span></span> <span data-ttu-id="0d62b-117">Ésta es una estrategia peligrosa debido a que los clientes normalmente liberan un objeto cuando su recuento de referencia llega a cero.</span><span class="sxs-lookup"><span data-stu-id="0d62b-117">This is a dangerous strategy because clients typically free an object when its reference count reaches zero.</span></span> <span data-ttu-id="0d62b-118">Liberación de la memoria para el objeto de receptor advise haría que el puntero _this_ no válido.</span><span class="sxs-lookup"><span data-stu-id="0d62b-118">Freeing the memory for the advise sink object would render the  _this_ pointer invalid.</span></span>
