<span data-ttu-id="6ab4d-105">Los proveedores de almacén de mensajes pueden combinarse estrechamente con un proveedor de transporte.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-105">Message store providers can be tightly coupled with a transport provider.</span></span> <span data-ttu-id="6ab4d-106">Acoplar estrechamente los proveedores de servicios MAPI significa implementar los dos proveedores de modo que el proveedor de almacenamiento y el proveedor de transporte puedan comunicarse para que el proceso de envío y recepción de mensajes sea más eficaz.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-106">Tightly coupling MAPI service providers means implementing the two providers such that the store provider and transport provider can communicate to make the process of sending and receiving messages more efficient.</span></span> <span data-ttu-id="6ab4d-107">La ventaja de hacerlo es que las mejoras de rendimiento pueden producirse cuando dos proveedores de servicios pueden interactuar entre sí directamente en lugar de hacerlo mediante la cola MAPI.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-107">The benefit of doing this is that performance improvements can result when two service providers can interact with each other directly rather than by means of MAPI spooler.</span></span> <span data-ttu-id="6ab4d-108">Para une estrechamente un proveedor de almacén de mensajes a un proveedor de transporte, el proveedor de transporte debe colocar el identificador de entrada del proveedor del almacén de mensajes en la propiedad **PR_OWN_STORE_ENTRYID** ([PidTagOwnStoreEntryId](pidtagownstoreentryid-canonical-property.md)) en la fila del proveedor de transporte en la tabla de estado MAPI.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-108">To tightly couple a message store provider to a transport provider, the transport provider must place the message store provider's entry identifier in the **PR_OWN_STORE_ENTRYID** ([PidTagOwnStoreEntryId](pidtagownstoreentryid-canonical-property.md)) property in the transport provider's row in the MAPI status table.</span></span> <span data-ttu-id="6ab4d-109">Esto permite que la cola MAPI conecte el proveedor de almacenamiento al proveedor de transporte.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-109">This enables MAPI spooler to connect the store provider to the transport provider.</span></span>
  
<span data-ttu-id="6ab4d-110">No hay ningún requisito de que un proveedor de almacén de mensajes esté estrechamente unido a cualquier otro proveedor de servicios.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-110">There is no requirement that a message store provider ever be tightly coupled with any other service provider.</span></span> <span data-ttu-id="6ab4d-111">El proveedor de servicios más común para asociar estrechamente con un proveedor de almacén de mensajes es un proveedor de transporte.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-111">The most common service provider to tightly couple with a message store provider is a transport provider.</span></span> <span data-ttu-id="6ab4d-112">Esto suele hacerse para que el envío y la recepción de mensajes se puedan realizar sin que se incluya la cola MAPI.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-112">This is usually done so that sending and receiving messages can be accomplished without involving the MAPI spooler.</span></span> <span data-ttu-id="6ab4d-113">Por ejemplo, cuando el usuario envía un mensaje saliente, el proveedor de almacenamiento de mensajes combinado y el proveedor de transporte pueden enviarlo directamente.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-113">For example, when the user submits an outgoing message, the combined message store provider and transport provider can send it directly.</span></span> <span data-ttu-id="6ab4d-114">Los proveedores de servicios combinados no tienen que notificar primero a la cola MAPI que hay un nuevo mensaje que class="sxs-lookup"><span data-stu-id="6ab4d-114">The combined service providers do not have to first notify MAPI spooler that there is a new message to process and then wait for MAPI spooler to initiate the process of transferring the message from the message store provider to the transport provider.</span></span> <span data-ttu-id="6ab4d-115">Esto tiene ventajas particulares cuando se usa un almacén de mensajes basado en servidor minimizando el tráfico de red entre el equipo del usuario y el servidor.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-115">This has particular benefits when a server-based message store is being used by minimizing network traffic between the user's computer and the server.</span></span>
  
<span data-ttu-id="6ab4d-116">En general, no hay procedimientos bien especificados para los proveedores de servicios de acoplamiento estricto.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-116">In general, there are no well-specified procedures for tightly coupling service providers.</span></span> <span data-ttu-id="6ab4d-117">Sin embargo, debe usar las siguientes directrices:</span><span class="sxs-lookup"><span data-stu-id="6ab4d-117">However, you should use the following guidelines:</span></span>
  
- <span data-ttu-id="6ab4d-118">Si el motivo para acoplar estrechamente a los proveedores de servicios es el rendimiento, tenga en cuenta que el acoplamiento quita partes del subsistema MAPI de los procesos en los que normalmente estarían implicadas esas partes.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-118">If the reason for tightly coupling service providers is performance, be aware that the coupling takes parts of the MAPI subsystem out of the processes that those parts would normally be involved in.</span></span> <span data-ttu-id="6ab4d-119">Esto implica que las partes individuales del proveedor de servicios combinado deben interactuar entre sí de forma que simula la interacción que normalmente tendrían con las partes del subsistema MAPI que no se están utilizando.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-119">This implies that the individual parts in the combined service provider should interact with each other in a way that simulates the interaction they would normally have with the parts of the MAPI subsystem that are not being used.</span></span>
    
- <span data-ttu-id="6ab4d-120">Cuando los proveedores de servicios estrechamente acoplados interactúan con otros componentes MAPI, aún deben interactuar con ellos de la manera exacta que lo harían si no estuvieran estrechamente acoplados.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-120">When tightly coupled service providers do interact with other MAPI components, they must still interact with them in exactly the way they would if they were not tightly coupled.</span></span> <span data-ttu-id="6ab4d-121">Por ejemplo, si un usuario usa un proveedor de almacenamiento de mensajes combinado y un proveedor de transporte como su almacén de mensajes predeterminado, pero usa un proveedor de transporte independiente para enviar mensajes, como puede ocurrir cuando un usuario toma un equipo en la carretera y cambia a un proveedor de transporte remoto, la parte del almacén de mensajes del proveedor de servicios estrechamente acoplado aún debe interactuar con la cola MAPI como si se tratase de un proveedor de almacén de mensajes independiente.</span><span class="sxs-lookup"><span data-stu-id="6ab4d-121">For example, if a user is using a combined message store provider and transport provider as their default message store but is using a separate transport provider to send messages — as can happen when a user takes a computer on the road and switches to a remote transport provider — the message store portion of the tightly coupled service provider must still interact with MAPI spooler just as if it were a standalone message store provider.</span></span>
